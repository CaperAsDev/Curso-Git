Git

Git es nuestro sistema de control de versiones:

  • Podemos hacerle seguimiento a un archivo o carpeta. Este seguimiento nos permite volver a versiones pasadas sin problema.
  • Podemos crear ramas para trabajar en diferentes dimensiones nuestro archivo sin dañar el archivo base.
  • Estas ramas las podemos fusionar a la rama principal cuando queramos y asi aplicar los cambios.

Para manejar git usamos nuestra terminal o linea de comandos. Existe una gran cantidad de comandos pero algunos de los mas usados son:

  • Init nos permite iniciar un repositorio, este sera el lugar donde se guarden todos estos cambios y versiones.
  • Add : es el comando que lleva los archivos que le indiquemos al staging, este es el espacio en memoria RAM que se dedica a los cambios que aun no han sido enviados al repositorio.
  • Commit : es el comando que envia esos archivos que estaban en staging al repositorio que inciamos con init. Para hacer commit debemos usar un atributo "-m" y escribir un mensaje que nos indicara a que cambios corresponde el commit.
  • Log : nos permite ver el historial de commit, cada commit tiene un codigo y con el podemos con ayuda de otro comando volver a la version de un archivo en ese commit.
  • Diff : con este comando veremos la diferencia entre dos versiones del mismo archivo, si no le ponemos ningun codigo comparara las dos ultimas versiones, si queremos comparar alguna en especifico debemos darle los dos codigos de commit al comando(separados por un espacio)
  • Status : con este comando veremos en que estado se encuentran nuestros archivos, veremos si hay cambios para hacer commit o archivos que no estan tracked (en seguimiento) o archivos que esten en staging esperando su commit.
  • rm : asi como en la terminal, rm significa remove y lo que hace es borrar un archivo de git, asi que se le dejara de hacer seguimiento.
  • Reset : este comado regresa un archivo al commit indicado, se puede usar con --soft, -mixed o --hard y cada uno sera mas extremo con el regreso de version: el hard eliminara todos los commit sucedidos luego del momento al que queremos regresar y elminara tambn la informacion en el area de staging, y el soft regresara pero mantendra en el staging la informacion que se encuentre ahi, permitiendonos aplicar los cambios desde la version a la que regresamos(es como hacer que los otros commits no existieron y cambiar todo desde un commit especifico).
  • Checkout : te permite viajar en el tiempo como reset solo que con la posibilidad de volver al futuro. nada mas usa el codigo del commit al que quieres volver y para regresar a donde estabas usa la palabra "master" en lugar del codigo (o el nombre de la rama principal). Con checkout creas nuevas ramas para experimentar y hacer cosas sin modificar la rama principal.
  • Show: este comando me mostrata informacion del ultimo commit realizado o si especificamos el commit-id luego de show veremos los cambios; adiciones y sustracciones de los archivos en ese commit.
  • log --stat: con este comando vere la cantidad de insertions(adiciones) y detetion(eliminaciones) que hice en los diferentes commit a parte del espacio nuevo ocupado.

Cuando trabajas en un proyecto con otras personas, van a necesitar que todos tengan los mismo archivos y que se esten actualizando constantemente con los avances que genera cada uno, para esto se usa un repositorio remoto donde puedes trabajar con muchas personas y que todas esten aportando simultaneamente. Para trabajar con repositorios remotos necesitaras algunos comandos adicionales:

  • git clone url_del_servidor_remoto: nos permite descargar los archivos de la última versión de la rama principal y todo el historial de cambios en la carpeta .git.
  • git push: luego de hacer git add y git commit debemos ejecutar este comando para mandar los cambios al servidor remoto
  • git fetchLo usamos para traer actualizaciones del servidor remoto y guardarlas en nuestro repositorio local (en caso de que hayan, por supuesto).
  • git merge: también usamos el comando git merge con servidores remotos. Lo necesitamos para combinar los últimos cambios del servidor remoto y nuestro directorio de trabajo.
  • git pull: básicamente, git fetch y git merge al mismo tiempo.

A continuacion un lista con unos comando mas especificos que igualmente son utiles dependiendo de la circunstancia.

  • git log --online: Te muestra el id commit y el título del commit.
  • git log --decorate: Te muestra donde se encuentra el head point en el log.
  • git log --stat: Explica el número de líneas que se cambiaron brevemente.
  • git log -p: Explica el número de líneas que se cambiaron y te muestra que se cambió en el contenido.
  • git shortlog: Indica que commits ha realizado un usuario, mostrando el usuario y el título de sus commits.
  • git log --graph y git log --pretty=format:"%cn hizo un commit %h el dia %cd": muestra mensajes personalizados.
  • git log -3: Limitamos el número de commits
  • git log --after="2022-6-8"
  • git log --after="today"
  • git log --after="2022-6-8" --before="today": Estos tres ubican commits por fechas.
  • git log --author="Name Author": Commits echos por quien tenga exactamente el nombre indicado.
  • git log --grep="ABC": Busca que conicidan con exactamente el mensaje puesto entre comillas.
  • git log --grep="ABC" -i: Busca los commits ignorando si es mayuscula o minuscula

Las ramas son la forma de hacer cambios a nuestros archivos sin comprometer la rama primcipal, al trabajar en ramas secundarias tenemos varias ventajas:

  • Pueden haber personas trbajando simultaneamente en diferentes ramas, esto permite un progreso mas fluido del proyecto.
  • Podemos hacer cambios experimentales al proyecto sin afectar la rama principal, asi, si funcionan los cambios podremos fusionar las ramas y aplicar los cambios.

Head resulta ser como nuestro indicador de la ubicacion en donde estamos en el proyectos segun nuestros commits en git, por defecto el head se ubucara en donde se realizo el ultimo commit, pero con el comando checkout nos podemos mover entre ramas.Algunos comandos utiles a la hora de trabajar con ramas son:

  • git branch branchName: este comando crea una nueva rama con el nombre que se indique. Las ramas se crean desde el lugar donde estemos ubicados.
  • git branch -b branchName:este comando a parte de crear la rama te dirige a ella, ubicando el head en la misma a diferencia del primer comando donde solo la crea y debes hacer un checkout para dirigir el head a ella.
  • git checkout branchName/commit-id:usar este comando nos llevara al commit o a la rama indicada.
  • git reset commit-id:este comando nos lleva a cualquier commit sin importar la rama, sin embargo borrara todo commit posterior. Revisar opciones de reset en el parrafo anterior.
  • git commit -am:este comando hace git add y git commit -m, solo debemos poner el mensaje a continuacion entre comillas y listo. Solo funciona con archivos que ya tengan historial.
  • git branch -l:este comando listara todas las ramas que existen.
  • git branch -m branchName branchNewNameeste comando nos permite cambiar de nombre una rama.

Cuando se van a fusionar ramas usamos el comando git merge nombre_de_la_rama_a_fusionar, el merge siempre se hace trayendo los cambios de la rama que estamos llamando a la rama donde estamos, si estamos en la rama master y traemos los cambios de otra rama, el resultado sera un commit donde nuestra rama master ahora incuye los cambios de la otra rama. En caso de presentarse conflictos(la misma linea de codigo modificada en ambas ramas) se debera solucionar borrando el cambio que no se quiere y terminando el commit.

GitHub

Han pasado varias cosas desde lo ultimo que estuve probando. Ahora estamos uniendo un repositorio local con un repositorio remoto en Git Hub. Antes, cuando hacia git clone con la url del repositorio remoto, lo que pasaba era que me creaba la carpeta del repositorio en mi carpeta local, asi, si yo queria subir las cosas a github tenia que mover los archivos a esa carpeta que me creo... Creo que lo mas sencillo a hacer desde un inicio es crear primero el repositorio remoto y clonarlo al local (es menos laborioso).

Yo ya tenia estor archivos HTML donde estoy escribiendo y el css de estilos en mi pc trackeados y demas, pero al quererlos subir a github tuve que hacer varios comandos que numerare a continuacion.

  • git remote add origin (url):este es el primer comando a ejecutar para unir los repositorios, en este paso pasaron varias cosas: una de ellas fue que no debi copiar la url sino la ssh; esto me ubiera ahorado un dolor de cabeza al tener que autentificar mi cuenta ya que para hacer el push luego me pediria cuenta y contraseña de github, pero git hub ya no acepta este tipo de confirmacion, debe ser con un token y para eso se usa ssh, asi que tuve que instalar otro complemento que me manejara ese proceso de autentificacion, para instalarlo use "brew install gh" y segui las instrucciones.
  • git pull origin(la rama origing remota) master/main(la rama local) --allow-unrelated-histories:este comando se usa ya que al ser diferentes repositorios con diferentes historias debes permitir juntarlos(trael el repositorio remoto al local), a mi me salieron otros problemas que soluciones siguiendo los comandos que me decia la terminal, era algo de hacer merge o rebase (la diferencia en lo que entendi es que el merge respeta y mantiene la historia de las dos ramas y el rebase borra el historial de la que se junta)
  • git push origin(manda cambios a orignin remoto) master(desde rama master o main local): con este comando enviamos los commit desde el repositorio local al repositorio remoto, nuestro problema es que nuestra rama se llama master y la rama de github se llama main asi que al no coincidir github interpreta que estamos creando una nueva rama y envia los cambios a una rama de main llamada master. Para solucionar esto hay que cambiarle el nombre por defecto a los nuevos repositorios locales con el comando "git config --global init.defaultBranch main", eso servira para los nuevos, para el actual o hacemos merge cada que queramos de la rama master que se crea o cambiamos el comando de push de tal manera: "git push origin master:main": de esta forma le decimos que haga push a la rama origin con los datos de nuestra rama master a su rama main.
